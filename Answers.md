2. Stack choice

I selected Next.js App Router with TypeScript because this project required both frontend UI rendering and backend API orchestration in a single application.

Next.js Route Handlers allowed me to:

keep API keys server-side
centralize orchestration logic
handle validation and partial failures cleanly
avoid maintaining separate frontend/backend deployments

TypeScript improved:

API contract safety
maintainability
safer refactoring
predictable data flow

TailwindCSS was chosen because it allowed rapid UI iteration and responsive dashboard styling without writing large custom CSS files.

A worse choice would have been a purely frontend React application directly calling third-party APIs from the browser because:

API keys would be exposed
orchestration logic would become scattered
upstream API failures would be harder to normalize
centralized validation and graceful degradation would be weaker
3. One real edge case
Edge Case

The application handles partial weather API failures without failing the full comparison response.

File
src/app/api/compare/route.ts
Relevant Section

The Promise.allSettled() weather orchestration block.

Why It Matters

If one weather request fails:

country information is still returned
warnings are generated
the frontend still renders usable comparison data

Without this handling:

a single upstream weather failure would crash the entire comparison request
the user would receive no usable output

This implements graceful degradation instead of complete request failure.

One example where I changed AI-generated output:
The original weather API URL generated by AI accidentally included an extra comma after the API key:

&appid=${API_KEY},&units=metric

I debugged and corrected it to:

&appid=${API_KEY}&units=metric

because the extra comma caused OpenWeather authentication failures.

I also modified several frontend styling suggestions to better match the final dashboard layout and spacing system I wanted.

5. Honest gap

One weakness in the current implementation is that there is no caching layer for external API requests.

Currently:

every comparison triggers live external API calls
repeated requests for the same countries are not optimized

With another day, I would:

add server-side caching
implement request deduplication
reduce API latency and rate-limit pressure

I would likely use:

Next.js fetch caching
or Redis for production-grade caching behavior
